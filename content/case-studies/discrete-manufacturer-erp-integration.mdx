## The situation

Job-shop manufacturing runs on sequence and priority. When a customer calls to expedite an order, the planner adjusts priority in SAP. That change used to take 4 hours to reach the floor — because the floor only got updates when a supervisor printed a new job list.

Meanwhile, SAP had no idea which jobs were actually running, which were waiting for material, and which were done. That data lived in the heads of the work-centre operators.

## What we built

### The middleware

SAP exposes work order events via RFC (Remote Function Calls). We built a Node.js service that subscribes to those events — new releases, priority changes, cancellations — and translates them into structured messages delivered to a PostgreSQL database the floor application reads from.

The translation layer is the entire value. SAP speaks SAP. The floor speaks operational reality. The middleware maps between them without either side needing to change.

### The floor application

A React application runs on a shared screen at each work centre. It shows the operator their current queue, sorted by priority. When a job's priority changes in SAP, the queue re-sorts within 15 seconds — no printout, no supervisor walk.

Operators confirm job start and completion through the same screen. That event fires back through the middleware and updates the work order status in SAP.

### What we didn't build

No MES. No full WIP tracking system. No barcode scanners. The scope was a reliable bidirectional bridge between SAP and the floor — nothing more. Adding complexity would have added failure points.

## Outcome

Paper routing eliminated on day one of go-live. The first priority escalation that came through SAP reached the floor in 12 seconds. The operations manager described it as "the first time in 20 years I know what's actually happening on the floor right now."

The 18% lead time reduction came from eliminating queue ambiguity — operators were no longer working from a stale printout that didn't reflect the current priority.
